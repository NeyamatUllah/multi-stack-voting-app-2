# Architecture — Multi-Stack Voting App

## Overview

A distributed voting application deployed on AWS. Users cast votes through a Python web app, votes flow through Redis into a .NET worker, which persists them in PostgreSQL. A Node.js app reads from PostgreSQL and displays live results.

---

## Application Stack

| Service  | Language / Runtime | Docker Image           | Port |
|----------|--------------------|------------------------|------|
| Vote     | Python (Flask)     | `neyamat/vote:latest`  | 8080 |
| Result   | Node.js            | `neyamat/result:latest`| 8081 |
| Worker   | C# (.NET 8)        | `neyamat/worker:latest`| —    |
| Redis    | —                  | `redis:alpine`         | 6379 |
| PostgreSQL | —               | `postgres:15-alpine`   | 5432 |

### Data Flow

```
User (browser)
     │
     ▼
  Vote App (Flask, :8080)
     │  writes vote to Redis
     ▼
  Redis (:6379)
     │  Worker polls Redis
     ▼
  Worker (.NET 8)
     │  persists vote to PostgreSQL
     ▼
  PostgreSQL (:5432)
     │  Result app reads counts
     ▼
  Result App (Node.js, :8081)
     │
     ▼
User (browser)
```

## AWS Infrastructure

### Region & VPC

- **Region:** `us-east-1`
- **VPC CIDR:** `10.0.0.0/16`

### Subnets

| Subnet         | CIDR          | Type    | Hosted On         |
|----------------|---------------|---------|-------------------|
| public         | 10.0.1.0/24   | Public  | Bastion, Instance A |
| private-b      | 10.0.2.0/24   | Private | Instance B        |
| private-c      | 10.0.3.0/24   | Private | Instance C        |

### EC2 Instances

| Instance   | Name Tag                    | Subnet   | Services              | Type     |
|------------|-----------------------------|----------|-----------------------|----------|
| Bastion    | voting-app-bastion          | Public   | SSH entry point only  | t3.micro |
| Instance A | voting-app-A-frontend       | Public   | Vote (:8080) + Result (:8081) | t3.micro |
| Instance B | voting-app-B-redis-worker   | Private  | Redis (:6379) + Worker | t3.micro |
| Instance C | voting-app-C-postgres       | Private  | PostgreSQL (:5432)    | t3.micro |

- **AMI:** Ubuntu 22.04 LTS (Canonical, `ubuntu-jammy-22.04-amd64`)
- **Key pair:** `voting-app-key` (`~/.ssh/voting-app-key.pem`)
- Docker CE is installed on A, B, and C via Ansible

### Network Topology

```
Internet
   │
   ├──► Bastion (public, SSH :22)
   │
   ├──► Instance A (public, :8080, :8081)
   │         │
   │    [private network]
   │         │
   │         ├──► Instance B (10.0.2.x, Redis :6379 + Worker)
   │         │         │
   │         │    [private network]
   │         │         │
   │         └──► Instance C (10.0.3.x, PostgreSQL :5432)
   │
   └── NAT Gateway (public subnet) ──► Private instances outbound
```

### Internet Gateway & NAT Gateway

| Component        | Purpose                                              |
|------------------|------------------------------------------------------|
| Internet Gateway | Inbound/outbound traffic for public subnet          |
| NAT Gateway      | Outbound-only internet access for private instances (pulling Docker images) |
| Elastic IP       | Attached to NAT Gateway — static public IP          |

## Security Groups (Least Privilege, SG-to-SG)

| Security Group   | Inbound Rules                                                                 |
|------------------|-------------------------------------------------------------------------------|
| `bastion-sg`     | Port 22 from `0.0.0.0/0`                                                     |
| `vote-result-sg` | Port 8080/8081 from `0.0.0.0/0`; Port 22 from `bastion-sg`                  |
| `redis-worker-sg`| Port 6379 from `vote-result-sg`; Port 22 from `bastion-sg`                  |
| `postgres-sg`    | Port 5432 from `redis-worker-sg` + `vote-result-sg`; Port 22 from `bastion-sg` |

All security groups allow unrestricted outbound traffic.

> SG-to-SG rules are used instead of CIDR blocks so rules remain valid when IPs change after `terraform destroy` / `apply`.

## SSH Access Pattern

All SSH into the cluster goes through the Bastion host using ProxyJump:

```
Local machine
     │  SSH :22
     ▼
Bastion (public IP)
     │  ProxyJump → private IP
     ├──► Instance A (10.0.1.x)
     ├──► Instance B (10.0.2.x)
     └──► Instance C (10.0.3.x)
```

`~/.ssh/config` is auto-generated by `infra/scripts/update-ssh-config.sh` after every `terraform apply`.

## Terraform State Management

| Resource           | Purpose                              |
|--------------------|--------------------------------------|
| S3 bucket          | `voting-app-tfstate-*` — remote state storage (versioned, AES256 encrypted) |
| DynamoDB table     | `voting-app-tf-locks` — state locking (prevents concurrent applies) |

- Bootstrap state (`infra/bootstrap/terraform.tfstate`) is **local by design** — do not re-apply.
- Main infra state (`infra/terraform/`) is stored remotely in S3.

## Operations

### Prerequisites

```bash
# Verify Terraform
terraform -version

# Verify AWS CLI
aws sts get-caller-identity

# Create EC2 key pair (once)
aws ec2 create-key-pair \
  --key-name voting-app-key \
  --query 'KeyMaterial' \
  --output text > ~/.ssh/voting-app-key.pem
chmod 400 ~/.ssh/voting-app-key.pem

# Install Ansible
sudo apt update && sudo apt install -y ansible
```

### Bootstrap (run once only)

Sets up the S3 bucket and DynamoDB table for remote Terraform state. State is kept **local** — do not re-apply.

```bash
cd infra/bootstrap
terraform init
terraform apply
# Copy the S3 bucket name from the output and put it in infra/terraform/backend.tf
```

### Provision Infrastructure

```bash
cd infra/terraform
terraform init   # only needed if .terraform/ is missing
terraform plan
terraform apply
```

### Refresh SSH Config & Ansible Vars (after every apply)

IPs change after every `terraform destroy` + `apply`. This script reads the new IPs from `terraform output` and updates `~/.ssh/config` and `group_vars/`.

```bash
cd ~/ironhack-bootcamp/practice-projects/multi-stack-voting-app-2
chmod +x infra/scripts/update-ssh-config.sh   # once
./infra/scripts/update-ssh-config.sh
```

Verify with: `ssh backend-instance-1` and `ssh db-instance-1`

### Deploy Containers

```bash
ansible-playbook -i infra/ansible/inventory.ini infra/ansible/playbook.yml
```

If the result container fails to come up on a fresh instance:

```bash
ssh frontend-instance-1 'sudo docker restart result'
```

### Verify

- **Vote app:** `http://<Instance A public IP>:8080` — cast a vote, tick should appear
- **Result app:** `http://<Instance A public IP>:8081` — vote count appears bottom-right

### SSH Access

```bash
# Direct (ProxyJump configured automatically by update-ssh-config.sh)
ssh bastion
ssh frontend-instance-1
ssh backend-instance-1
ssh db-instance-1

# Manual (before running the script)
ssh -i ~/.ssh/voting-app-key.pem ubuntu@<bastion_public_ip>
ssh -i ~/.ssh/voting-app-key.pem -J ubuntu@<bastion_public_ip> ubuntu@<instance_private_ip>
```

### Tear Down

```bash
cd infra/terraform
terraform destroy
```

> The NAT Gateway and EC2 instances cost money while running — always destroy when done.
> Known issue: `terraform destroy` may fail to delete subnets if a VPC endpoint/ENI exists. Delete the endpoint manually in the AWS console first.

## Add-Ons Implemented

| Add-On                  | Detail                                                       |
|-------------------------|--------------------------------------------------------------|
| Dedicated Bastion Host  | Separate EC2 + `bastion-sg`; all SSH routes through it      |
| SG-to-SG Security Rules | No hardcoded CIDRs for internal traffic                     |
| Remote Terraform State  | S3 + DynamoDB with encryption and versioning                |
| PostgreSQL Named Volume | `-v postgres-data:/var/lib/postgresql/data` — data survives container restarts |
